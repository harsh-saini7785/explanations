## ♿ Accessibility (a11y) in Frontend Development

Accessibility (a11y) refers to building web applications that can be used by everyone, including people with visual, motor, hearing, or cognitive disabilities. It ensures that users relying on assistive technologies like screen readers such as NVDA or VoiceOver, or those who navigate using keyboards instead of a mouse, can effectively interact with the application. The goal of accessibility is to create inclusive user experiences by following four core principles: content should be perceivable (users can see or hear it), operable (users can interact with it via keyboard or other inputs), understandable (clear and predictable UI), and robust (compatible with assistive technologies).

Improving accessibility starts with using semantic HTML elements like `<button>`, `<form>`, `<label>`, and `<ul>` instead of generic `<div>` elements, as these provide built-in accessibility features such as keyboard navigation and screen reader support. Ensuring full keyboard accessibility is critical, allowing users to navigate and interact using keys like Tab, Enter, and Arrow keys. Proper labeling of form inputs using `<label>` or `aria-label` helps screen readers convey the purpose of inputs clearly. For dynamic applications (such as React apps), ARIA attributes like `aria-live` are used to notify screen readers of content updates without requiring a page reload. Providing meaningful button text or accessible labels for icon-only buttons (e.g., delete icons) improves usability for assistive technologies.

Focus management is another key aspect, where focus should be programmatically moved (e.g., returning focus to an input after adding a todo) to guide users effectively. Developers should avoid relying solely on visual cues like color to convey information and instead include text or icons for clarity. Proper document structure using headings (`<h1>`, `<h2>`) and lists (`<ul>`, `<li>`) enhances navigation for screen readers. Images must include descriptive `alt` text so users understand visual content. While ARIA attributes (like `aria-label`, `aria-live`, `aria-hidden`, etc.) are powerful, they should only be used when native HTML cannot achieve the desired accessibility, following the rule: “Prefer native HTML over ARIA.”

In practice, accessibility can be tested using keyboard navigation, screen readers, and tools like browser Lighthouse audits or accessibility testing extensions. Building accessible applications not only improves usability for all users but is also important for legal compliance and is a highly valued skill in frontend interviews.
